# 保險業 PII 數據脫敏深度作法研究

在保險業中，身分證字號、保單號碼、聯絡資訊等個人識別資訊（PII）是最高等級的保護對象。本研究探討如何在開發生命週期（SDLC）的早期階段攔截並脫敏這些數據。

## 1. 偵測機制：Regex + 敏感詞庫雙重過濾

實作雙層偵測邏輯以平衡效能與準確度：

### A. Regex 偵測模式
針對格式化數據進行初步掃描：
- **台灣身分證 (ID)**: `^[A-Z][12]\d{8}$`
- **手機號碼**: `^09\d{8}$`
- **保單號碼 (範例)**: `^PL\d{10}$` (依企業規範調整)
- **Email**: 常用標準 Regex

### B. 敏感詞庫 (Dictionary-based)
針對非結構化數據：
- **地址**: 透過縣市、路名關鍵字匹配。
- **姓名**: 結合常見姓氏庫與 NLP 實體識別（NER）技術。

## 2. 攔截層次設計

### A. IDE 端：本地代理（Local Proxy / Interceptor）
針對 IDE（如 Visual Studio, VS Code）與 GitHub Copilot 溝通的流量：
- **機制**: 設定 IDE 的 HTTP 代理至本地轉發器。
- **功能**: 在請求傳送到雲端前，即時掃描 Prompt 內容。若偵測到 PII，則自動進行屏蔽（Masking）或替換為虛擬數據。

### B. Git 端：Pre-commit Hook
防止開發者不慎將包含 PII 的測試數據或註解提交至版本控制系統：
- **實作**: 使用 `Husky` 或 Python `pre-commit` 框架。
- **邏輯**: 掃描 Staged 檔案內容，若違規則中斷 Commit 並告警。

## 3. 脫敏策略（De-identification）

- **靜態代碼與註解**: 採用「不可逆遮蔽」（如：`A123****89`）。
- **動態請求**: 採用「暫時性 Token 化」，確保模型能理解結構但無法獲取真實數據。

## 4. 落地建議

建議保險企業採取「先偵測告警，後自動阻斷」的階段性導入策略，並定期更新敏感詞庫以應對新的業務需求。
