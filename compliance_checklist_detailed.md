# 保險業導入 GitHub Copilot 資安合規注意事項 (依據自律規範)

本清單根據《保險業辦理資訊安全防護自律規範》整理，旨在協助保險公司在符合監管要求的前提下順利導入 GitHub Copilot。

## 一、 帳號與權限控管 (對應自律規範第 5 條)
*   **[一人一號原則]**：嚴格執行一人一號，禁止共享 Copilot 授權席位。
*   **[強制雙因子認證 (MFA)]**：GitHub Enterprise 必須強制開啟 MFA，建議整合企業現有的身分驗證器。
*   **[SSO 同步註銷]**：必須與企業 AD/LDAP 整合（如透過 Keycloak），確保員工離職或調職時，其 Copilot 存取權限能在 24 小時內自動失效。
*   **[最小權限分配]**：僅針對開發相關人員分配 Copilot 席位，並定期（至少每季）清查授權名單之合理性。

## 二、 個人資料保護 (對應自律規範第 11, 16 條)
*   **[數據脫敏 (Data Masking)]**：嚴格禁止將包含客戶姓名、身分證字號、保單號碼、聯絡電話等 PII 的代碼或註解傳輸至雲端。
*   **[建立本地網關]**：建議實作中介層（如 Milk-Local），在代碼片段發送至 GitHub 前，自動透過 Regex 或 NLP 技術識別並替換敏感資訊。
*   **[不訓練模型條款]**：採購時必須確認合約中載明「不使用企業代碼進行模型訓練與改進」，以確保保險業核心邏輯不外流。

## 三、 系統開發與變更安全 (對應自律規範第 6, 14 條)
*   **[AI 代碼二次審核]**：所有 AI 產出的代碼片段必須視為「第三方委外產出內容」，嚴禁不經人工審查直接進入生產環境。
*   **[整合 SAST 掃描]**：AI 生成的代碼必須通過公司內部的靜態掃描工具 (如 SonarQube, CodeQL)，檢查是否存在 SQL Injection、Hardcoded Secrets 等安全漏洞。
*   **[版權與合規過濾]**：必須在 GitHub Enterprise 全域策略中強制開啟「Block suggestions matching public code」，避免引入具版權爭議的開源代碼。

## 四、 稽核軌跡與日誌管理 (對應自律規範第 16 條)
*   **[日誌保留 180 天]**：GitHub 提供之 Audit Logs 必須定期（建議每日）導出至企業內部日誌管理系統（如 Logstash/Splunk）保存，確保符合 180 天的法定保留期限。
*   **[異常行為監控]**：應監控 Copilot 使用量的異常激增或來自非授權 IP 的存取行為，並建立即時告警機制。

## 五、 委外與風險評估 (對應自律規範第 15 條)
*   **[供應商資安稽核]**：應將 GitHub 納入委外資安評估清單，索取其 SOC2 Type II 或相關 ISO 認證報告作為內部稽核附件。
*   **[定期災難演練]**：評估若 GitHub Copilot 服務中斷時，開發團隊的應變計畫（如切換至本地 LLM 備援節點）。

---
**🥛 Milk 的導入建議：**
建議採取「漸進式導入」：先從非核心系統的內部工具開發開始，並同步建立「保險業專用 Prompt 安全指南」，教育開發者正確且安全地與 AI 互動。
